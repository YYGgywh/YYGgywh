# 六爻排盘项目（初期版）开发说明书

# 一、项目概述

## 1.1 项目背景

本项目为六爻排盘工具类应用，初期聚焦六爻排盘核心功能，配套实现用户注册、登录及排盘记录管理功能，为用户提供便捷、稳定的排盘服务；后期可扩展奇门、八字、六壬等多类排盘功能及社交讨论模块，满足用户多样化需求。

## 1.2 项目目标

- 完成六爻排盘核心功能落地，确保排盘逻辑准确、运行流畅；

- 实现轻量化用户体系（注册、登录、排盘记录保存/查询），降低用户使用门槛；

- 采用轻量化技术栈，快速完成初期开发与部署，控制开发成本；

- 预留功能扩展空间，为后期新增多类排盘、社交功能奠定基础；

- 保障用户数据安全，优化交互体验，适配PC端与移动端。

## 1.3 适用范围

本说明书适用于本项目初期（小项目阶段）前端、后端开发人员，明确开发规范、技术选型、功能要求及交付标准，作为开发协作、进度管控、成果验收的依据。

# 四、核心功能开发要求

## 4.1 六爻排盘功能（已完成）

核心排盘逻辑已实现，前端需完成页面渲染、参数传入，后端需配合接收排盘参数、存储排盘结果，确保排盘准确性、运行流畅性，适配PC端与移动端响应式展示。

## 4.2 用户体系功能（核心开发项）

### 4.2.1 注册功能

- 支持手机号+验证码注册，可选密码设置（兜底）；

- 表单校验：手机号为11位数字，验证码为6位数字，密码≥6位（含字母+数字）；

- 验证码发送：点击发送后倒计时60秒，限制重复发送，初期控制台模拟发送，后期对接短信服务商；

- 注册校验：手机号唯一，验证码有效（5分钟过期），需勾选用户隐私协议；

- 注册成功后，自动生成用户ID，返回成功提示，引导用户登录。

### 4.2.2 登录功能

- 支持两种登录方式：手机号+验证码、手机号+密码；

- 登录校验：手机号已注册、验证码有效/密码正确；

- 登录成功后，生成JWT Token（有效期7天），前端存储Token与用户基础信息，后端存储Token并做有效性校验；

- 登录态维护：Token过期后，前端自动跳转登录页，提示用户重新登录；

- 登录成功后，检测前端本地临时排盘记录，弹窗提示用户是否同步至账号。

### 4.2.3 排盘记录管理功能

- 未登录状态：排盘结果存储至前端localStorage，提示用户登录可永久保存；

- 已登录状态：排盘完成后，自动调用后端接口保存排盘记录（关联用户ID）；

- 排盘记录查询：用户可在个人中心查询本人排盘记录，按创建时间倒序排列，支持分页（默认10条/页）；

- 数据存储：排盘记录需包含排盘类型、排盘参数、排盘结果、创建时间，支持后期多类排盘扩展；同时预留求占者补充说明字段，用于存储求占者对卦例的补充信息（如求占事项、背景说明等）；

- 评论功能：注册用户可对自己的排盘记录添加评论（如卦例解读、后续反馈等），评论信息关联排盘记录ID与用户ID，支持修改、删除自己的评论；

- 讨论预留：预留网友讨论模块数据存储与接口扩展空间，后期将实现类似小红书的公开卦例讨论功能（如点赞、评论、收藏、转发），支持注册用户对公开卦例进行交流互动。

# 五、数据库设计规范

## 5.1 数据库选型

初期采用SQLite数据库（轻量化、无需单独部署、开发成本低），适配小流量场景；后期用户量增长后，平滑迁移至MySQL数据库，保障高并发支持。

## 5.2 核心表结构

### 5.2.1 用户表（user）

|字段名|字段类型|是否必选|默认值|字段说明|
|---|---|---|---|---|
|id|INTEGER|是|自增|用户主键ID|
|phone|VARCHAR(11)|是|-|用户手机号，唯一索引|
|password|VARCHAR(255)|否|NULL|哈希后的用户密码（验证码登录可为空）|
|ext_info|TEXT|否|'{}'|扩展字段（JSON格式），预留会员、昵称等信息|
|create_time|INTEGER|是|当前时间戳|用户注册时间|
|update_time|INTEGER|是|当前时间戳|用户信息更新时间|
|is_active|TINYINT|是|1|用户状态（1：正常，0：禁用）|
### 5.2.2 排盘记录表（pan_record）

|字段名|字段类型|是否必选|默认值|字段说明|
|---|---|---|---|---|
|id|INTEGER|是|自增|排盘记录主键ID|
|user_id|INTEGER|是|-|关联用户ID，外键关联user表id|
|pan_type|VARCHAR(20)|否|'liuyao'|排盘类型（liuyao：六爻，预留qimen：奇门、bazi：八字等）|
|pan_params|TEXT|是|-|排盘参数（JSON字符串），存储起卦时间、生辰等信息|
|pan_result|TEXT|是|-|排盘结果（JSON字符串），存储卦象、爻辞等信息|
|create_time|INTEGER|是|当前时间戳|排盘记录创建时间|
|supplement|TEXT|否|NULL|求占者补充说明，存储求占事项、背景等信息|
### 5.3 索引设计

- 用户表：为phone字段建立唯一索引（idx_user_phone），确保手机号唯一，提升查询效率；

- 排盘记录表：为user_id与pan_type字段建立联合索引（idx_pan_user_type），提升按用户、排盘类型查询记录的效率；

- 新增评论表：为pan_record_id、user_id字段建立联合索引（idx_comment_pan_user），提升按排盘记录、用户查询评论的效率；为create_time字段建立索引（idx_comment_time），优化评论按时间排序的查询性能。

### 5.4 新增评论表（comment）

用于存储注册用户对排盘记录的评论信息，以及后期网友讨论相关数据，适配小红书式互动场景，结构如下：

|字段名|字段类型|是否必选|默认值|字段说明|
|---|---|---|---|---|
|id|INTEGER|是|自增|评论主键ID|
|pan_record_id|INTEGER|是|-|关联排盘记录ID，外键关联pan_record表id|
|user_id|INTEGER|是|-|关联评论用户ID，外键关联user表id|
|content|TEXT|是|-|评论内容（卦例解读、反馈、讨论等）|
|create_time|INTEGER|是|当前时间戳|评论创建时间|
|update_time|INTEGER|是|当前时间戳|评论更新时间|
|is_public|TINYINT|否|0|是否公开（0：仅自己可见，1：公开可讨论），预留网友讨论功能|
|ext_info|TEXT|否|'{}'|扩展字段（JSON），预留点赞数、回复数等网友讨论相关信息|
- 用户表：为phone字段建立唯一索引（idx_user_phone），确保手机号唯一，提升查询效率；

- 排盘记录表：为user_id与pan_type字段建立联合索引（idx_pan_user_type），提升按用户、排盘类型查询记录的效率。

# 六、接口开发规范

## 6.1 接口通用规范

- 接口前缀：统一为/api/v1/，便于后期版本升级与维护；

- 请求方式：GET（查询）、POST（新增/提交），遵循RESTful规范；

- 请求头：需登录的接口，需在请求头携带Token，格式为Authorization: Bearer {token}；

- 返回格式：统一为JSON，结构如下：
        `{
  "code": 200,  // 状态码（200：成功，400：参数错误，401：未登录/Token失效，500：服务器错误）
  "msg": "success",  // 提示信息
  "data": {}  // 业务数据（无数据时可为null）
}`

- 参数校验：所有接口需对传入参数进行校验，非法参数返回400状态码及明确提示。

# 八、安全规范

- 密码安全：用户密码必须使用bcrypt哈希存储，绝对禁止明文存储；

- Token安全：Token设置有效期，采用JWT加密，后端定期清理过期Token；

- 接口安全：所有需登录的接口必须进行Token校验，防止未授权访问；

- 参数安全：接口参数严格校验，防止SQL注入、XSS等攻击；

- 频率限制：验证码发送、登录/注册接口限制请求频率，防止短信轰炸、暴力破解。
